# 整体逻辑：

1. 获取题目。
2. 对每个题目：  
   a.  **代码生成**：尝试生成 C++ 代码。  
   b.  **本地测试与迭代优化**：  
   &emsp; i. 使用题目中的样例输入输出进行本地测试。  
   &emsp; ii. 如果测试失败，根据配置：  
   &emsp;&emsp; * 若 `CONFIG["only_gen_and_retry"]` 为 `True`，则放弃当前代码，回到 (a) 重新生成。  
   &emsp;&emsp; * 否则，进行**缺陷分析和代码修复**，然后回到 (b.i) 重新本地测试。  
   &emsp; iii. 如果本地测试达到最大次数仍失败，则放弃当前代码，回到 (a) 重新生成。  
   c.  **在线判题**：  
   &emsp; i. 如果本地测试通过，提交代码到 OJ。  
   &emsp; ii. 如果 OJ 判题结果不是 "正确"：  
   &emsp;&emsp; * 若 `CONFIG["only_gen_and_retry"]` 为 `True`，则放弃当前代码，回到 (a) 重新生成。  
   &emsp;&emsp; * 否则，进行**缺陷分析和代码修复**。  
   &emsp;&emsp; * 修复后，**必须再次通过本地测试** (回到 (b.i) 执行本地测试流程)。如果本地测试失败，则放弃当前代码，回到 (a)
   重新生成。  
   &emsp;&emsp; * 如果修复后本地测试通过，则回到 (c.i) 重新提交到 OJ。  
   &emsp; iii. 如果 OJ 判题本身失败（如网络错误），则重试在线判题。  
   &emsp; iv. 如果在线判题达到最大次数仍未 "正确"，则放弃当前代码，回到 (a) 重新生成。  
   d. 如果代码生成达到最大次数仍未成功解答题目，则该题目解答失败。
3. 并发处理多个题目。
4. 输出总结。

关于数据污染的测试：
通过向LLM提供不完整的题目信息，观察其是否能推测出完整的题目信息、选择正确的算法、生成正确的代码，来评估数据污染程度：  
设计了4个级别的数据污染程度：

| 级别 | 描述                           |
|----|------------------------------|
| 0  | 推测不出完整题目信息                   |
| 1  | 推测出完整题目信息，但无法选择正确的算法         |
| 2  | 推测出完整题目信息，选择正确的算法，但不能生成正确的代码 |
| 3  | 推测出完整题目信息，选择正确的算法，生成正确的代码    |

注：E/F两题的题目描述非常简洁，即使只保留前两句话和标题，也已经包含了完整的题目信息。同时，这两道题的算法选择也很直接，导致LLM出现了类似数据污染的表现。

关于模型选择与prompt工程的对比分析：  
在5月17日初赛后，通过分析比赛结果和与同学的讨论，我有以下发现：

1. 成功解答的题目的数量，主要取决于LLM的选型。OpenAI系列的LLM有非常明显的优势。
2. 我设计的复杂中文prompt，在应用于o4-mini和deepseek-v3时表现良好，但是在应用于gemini-2.5-pro-preview-05-06和deepseek-r1这两款think模型时，
   导致了严重的over think问题。
3. 有两位同学使用deepseek-r1，搭配简单的prompt，通过多次生成，就取得了和OpenAI系列LLM相同的AC数。这说明，在使用think模型时，应搭配简单的prompt，让LLM自主推理和发挥。

本次正式比赛后，我统计了480分以上的LLM Robot的模型、方法与prompt类型：

| ID | 模型                           | 方法         | prompt类型        | 得分  |
|----|------------------------------|------------|-----------------|-----|
| 19 | o4-mini                      | 多智能体       | 复杂英文prompt      | 621 |
| 14 | deepseek-r1                  | 多轮对话反馈OJ结果 | 简单中文prompt      | 548 |
| 12 | o3-mini-all                  | 多次生成       | 复杂英文prompt      | 533 |
| 06 | o4-mini                      | 本地测试+多次生成  | 简单中文prompt      | 498 |
| 20 | gemini-2.5-pro-preview-05-06 | 多次生成       | 简单英文prompt      | 493 |
| 16 | deepseek-r1                  | 多次生成       | 简单英文prompt+少量指令 | 487 |
| 15 | gpt-4.5-preview-2025-02-27   | 多轮对话反馈OJ结果 | 简单中文prompt      | 481 |

通过分析上表，可以得到以下结论：

1. OpenAI系列的LLM，以及deepseek-r1有明显的优势。
2. 对比19、12和06、15号Robot，虽然方法不同，但是AC的题目都是一次性直接生成的。可以认为，分数差异主要归因于复杂英文prompt能够为LLM设定明确的竞赛环境，同时减少了语言混淆的可能性。
3. 分析14号Robot的提交记录，同样发现了AC的题目都是一次性生成的现象。反馈机制存在两个部分有效样本：

| 题目 | 修复前正确率 | 修复后正确率 |
|----|--------|--------|
| C  | 28%    | 81%    |
| F  | 53%    | 67%    |

其中，第二个样本的有效性存在争议:  
(1)在之前的一轮反馈中，原本67%准确率的代码被优化后反而降至53%。该次准确率提升只是回调。  
(2)同样采用deepseek-r1的16号Robot直接AC了本题。 推测如果不进行多轮反馈，直接重新生成，也能得到更好的结果。

综合分析结论：

1. LLM解答竞赛级编程题的正确率，主要取决于LLM本身的能力（在排除数据污染的前提下）。选择代码和推理能力强大的LLM，比应用其他辅助方法更重要。
2. 复杂prompt对直接回答型LLM有帮助，但是会导致think模型over think。在使用think模型时，应搭配简单的prompt，让LLM自主推理和发挥。
3. LLM解答特定竞赛级编程题时的性能表现，呈现出明显的两极分化：要么一次生成直接AC；要么经过多次尝试和修改后也只能通过样例输入输出，甚至连样例都无法通过。
4. LLM对竞赛级编程题的代码进行故障定位和修复的能力，高度依赖于它正确解题的能力。换句话说，当LLM无法直接正确解答特定编程问题时，它也很难有效地调试与修复该问题相关的代码。
